Profilo GitHub

Panoramica

Ha progettato e implementato tutta la persistenza del progetto. Dal layer locale SQLite (9 tabelle, WAL mode, migrazioni idempotenti con backup pre-DROP) al layer cloud Supabase (schema PostgreSQL, Row Level Security, REST API client). È la base su cui poggia l'offline-first del gioco: il sistema funziona senza connessione, il cloud è un bonus.

Contributi per area

SQLite Locale

  • local_database.gd (831 righe) — autoload SQLite via godot-sqlite GDExtension v4.7
  • 9 tabelle progettate: accounts, characters, rooms, inventario, sync_queue, settings, save_metadata, music_state, placed_decorations
  • 7 indici su foreign keys per performance query
  • PRAGMA journal_mode=WAL + foreign_keys=ON + busy_timeout=5000
  • BEGIN/COMMIT/ROLLBACK transaction pattern in _on_save_requested
  • CRUD completo per ogni tabella: upsert, get, update, delete, soft_delete

Migrazioni Schema

  • Migration 1: characters legacy senza character_id → backup characters_bak + inventario_bak pre-DROP (safety net rollback-safe)
  • Migration 2: ALTER TABLE accounts ADD COLUMN per display_name, updated_at, password_hash, deleted_at, coins, inventario_capacita
  • Migration 3: ALTER TABLE inventario ADD COLUMN per item_type, is_unlocked, acquired_at
  • Migrations idempotenti — si possono eseguire più volte senza errori
  • Logging strutturato di ogni step migration per debugging

Auth + Security

  • auth_manager.gd (193 righe) — guest + username/password
  • PBKDF2-v2 hashing con salt per-account (16 bytes random), 10.000 iterazioni SHA-256
  • Legacy hash migration: al primo login con hash vecchio, si migra automaticamente al formato v2
  • Rate limiting: 5 tentativi falliti → lockout 300s
  • Soft delete account (flag deleted_at) per audit trail
  • Integrazione is_open() check prima di ogni operazione DB (fix A26)

Supabase Cloud

  • supabase_client.gd (499 righe) — REST API client asincrono
  • Schema cloud: 15 tabelle PostgreSQL con RLS auth.uid() = user_id su ogni policy
  • 5 tabelle push live: profiles, user_currency, user_settings, music_preferences, room_decorations
  • Session token cifrato con chiave derivata da device (ConfigFile.save_encrypted_pass) — fix B-019
  • HTTPS obbligatorio nella config validation (fix B-020)
  • Queue cap 500 per prevenire OOM (supabase_http.gd)
  • SQLite sync_queue persistente per retry offline
  • Graceful degradation su 404/400 relation error (schema evolving)

Data Mapping

  • supabase_mapper.gd — Single Point of Truth per local-to-cloud field mapping
  • Bidirectional: profile_to_cloud, cloud_profile_to_local, decorations_to_cloud, ecc.
  • .get(key, default) ovunque — missing field mai crash
  • supabase_config.gd — load config da user://config.cfg, validate url + anon_key

Data Integrity

  • HMAC-SHA256 save file validation design (architettura con Renan)
  • Atomic write pattern: temp file → rename per prevenire corruzione
  • Backup automatico prima di ogni save nuovo
  • Type validation ad ogni _apply_save_data (settings, music, inventory, character)
  • Clamping valori numerici: livello_stress 0-100, volume 0-1, capacita 1-999
